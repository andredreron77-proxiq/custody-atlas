We need to implement secure original-file access for Custody Atlas.

Goal:
Users should be able to view or download their original uploaded document securely, without exposing raw storage paths or using public URLs.

Scope:

1. Inspect the current document storage implementation

   * Determine:

     * where original files are stored
     * whether storage_path already exists and is reliable
     * whether Supabase Storage is being used directly
   * Do not assume; inspect the real code first

2. Add secure backend file access routes

   * Implement one or both of these:

     * GET /api/documents/:documentId/view
     * GET /api/documents/:documentId/download
   * Both routes must:

     * require authentication
     * verify document ownership server-side
     * verify case ownership if case_id is present
     * generate a short-lived signed URL (60–120 seconds preferred)
   * Do not expose raw storage paths to the client

3. Use private object access only

   * If the storage bucket/path is public, identify that clearly and switch to a private access pattern
   * Do not use public URLs for sensitive legal documents

4. Add UI entry points

   * On the Document Detail page, add:

     * “View original”
     * “Download original” (if feasible with current storage)
   * Keep the existing analysis/review page intact
   * Make these actions clearly separate from Ask Atlas actions

5. Add safe error handling

   * If the original file is missing or storage access fails:

     * show a clear user-facing message
     * log only documentId / userId / action
     * do not log document text or sensitive content

6. Add minimal development logging

   * Log:

     * signed URL generation success/failure
     * ownership verification success/failure
   * Do not log raw file contents or signed URLs

7. Keep everything additive and safe

   * Do not redesign the whole app
   * Do not break current document analysis or document-scoped Ask Atlas
   * Reuse the existing documents table metadata and storage path fields where practical

Important:

* Security and privacy are the priority
* Prefer short-lived signed URLs over direct file streaming for now
* Do not expose permanent URLs
* Do not trust client-provided storage paths
* Always look up the document record server-side first

At the end, show:

* files changed
* how original-file access works end-to-end
* how ownership is verified
* how long signed URLs last
* whether the storage bucket/path is private
* any remaining risks or follow-up recommendations
